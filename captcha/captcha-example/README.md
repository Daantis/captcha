# Набор server-driven CAPTCHA для sandbox

Пакет реализует 5 разных CAPTCHA под текущий Go SDK и песочницу:

- `odd-grid`: выбрать лишний элемент.
- `reality-swipe`: свайпнуть карточку как `возможно / невозможно`.
- `foreign-letter`: отметить буквы, не относящиеся к русскому алфавиту.
- `two-baskets`: разложить карточки по двум корзинам.
- `track-object`: проследить за фишкой после серверных перестановок.

## Как подключить

Один и тот же бинарь запускается с разными `CHALLENGE_ID`.

Поддерживаемые значения:

- `odd-grid`
- `reality-swipe`
- `foreign-letter`
- `two-baskets`
- `track-object`

В `main.go` билдер выбирает режим по `CHALLENGE_ID` и возвращает нужную реализацию `sdk/server.Captcher`.

## Быстрый запуск всего стенда

Из корня `sandbox-v3`:

```bash
docker compose up --build
```

Что поднимется:

- `sandbox` на `http://localhost:8080`
- `sandbox` gRPC на `localhost:50052`
- 5 отдельных CAPTCHA-инстансов:
  - `odd-grid`
  - `reality-swipe`
  - `foreign-letter`
  - `two-baskets`
  - `track-object`

После старта откройте `http://localhost:8080` и нажмите `Refresh`.

Если нужен ручной запуск без Docker для CAPTCHA, выставляйте:

```bash
TARGET_HOST=localhost:50052
HOST=localhost
CHALLENGE_ID=odd-grid
```

и запускайте `go run main.go` из `captcha/captcha-example`.

## Сборка фронта

Фронт лежит в `frontend/` и собирается в standalone HTML-файлы для embed:

```bash
npm install --no-package-lock
npm run build
```

Результат попадает в `internal/captcha/html/*.html`.

## Поведение по сложности

Используется единая динамическая шкала `1-100`.

- `1-20`: базовая механика, одна смысловая линия, минимум 5 пользовательских взаимодействий.
- `21-40`: добавляется второй смысловой прием внутри той же CAPTCHA.
- `41-59`: появляется смена правила по ходу сессии.
- `60-79`: минимум 2 подзадачи или сегмента внутри одной сессии.
- `80-99`: минимум 3 подзадачи и server-driven `PATCH` без действия пользователя.
- `100`: максимальная версия сценария с 4 подзадачами по профилю сложности.

## Что делает сервер

- не отдает полный ответ в HTML;
- хранит минимальное состояние сессии и одноразовые `entityId`;
- валидирует каждый клик / свайп / drag-drop;
- может отправлять `PATCH` без пользовательского ввода;
- очищает состояние по `HandleConnectionClosed` и `HandleCloseChallenge`.

## Что делает клиент

- после загрузки iframe отправляет `READY`;
- получает `INIT/PATCH/PROMPT/PROGRESS/RESULT` по бинарному протоколу;
- рендерит текущее состояние и отправляет компактные события обратно на сервер;
- ничего не вычисляет по правильности ответа локально.
