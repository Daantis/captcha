# Пакет для сдачи заказчику

## Что реализовано

Реализован набор из 5 отдельных server-driven CAPTCHA под текущий `sdk/server.Captcher`:

- `odd-grid`
- `reality-swipe`
- `foreign-letter`
- `two-baskets`
- `track-object`

Все механики работают через один Go-бинарь и выбираются по `CHALLENGE_ID`.

## Что соответствует ТЗ

- Задание не встраивается в HTML целиком: iframe получает только bootstrap-оболочку, а первое содержательное состояние приходит после `READY`.
- Клиент запрашивает задание через протокол и на каждом действии отправляет данные обратно на сервер.
- Сервер валидирует каждый `tap / swipe / drag-drop`, хранит только текущее состояние и выдает новые части задания по мере прохождения.
- На сложностях `80+` сервер отправляет unsolicited `PATCH` до действия пользователя.
- На сложностях `60 / 80 / 100` растет число подзадач: `2 / 3 / 4`.
- На всех типах низкая сложность остается понятной человеку за первые секунды.
- Клиент не знает полный ответ, не получает весь будущий сценарий и не вычисляет корректность локально.
- Стек минимальный: Go на сервере, standalone HTML + TypeScript/CSS на клиенте, без БД и внешних сервисов.

## Что показывать заказчику

Пороговые сценарии для демонстрации:

- `20`: простая версия без server push.
- `60`: появляется вторая подзадача или второй сегмент механики.
- `80`: появляются 3 подзадачи и server-driven обновления до действия пользователя.
- `100`: 4 подзадачи, максимальный вариант механики.

Хорошая демонстрация:

1. Поднять sandbox и все 5 CAPTCHA.
2. Показать `Refresh` в UI песочницы.
3. Открыть 2-3 разных типа CAPTCHA и пройти их на low/medium.
4. Отдельно показать любой сценарий на `80+`, где сервер сам меняет задание до действия пользователя.

## Как запускать локально

Из корня `sandbox-v3`:

```bash
docker compose down
docker compose up --build
```

После старта:

- UI sandbox: `http://localhost:8080`
- gRPC sandbox: `localhost:50052`

Дальше открыть UI и нажать `Refresh`.

## Что отдавать в production

Для production-интеграции достаточно:

1. Go-бинаря CAPTCHA.
2. Встроенных HTML-страниц из `internal/captcha/html`.
3. Настройки `TARGET_HOST`, `HOST`, `CHALLENGE_ID`.

Локальный `docker-compose.yml`, Dockerfile, frontend build tooling и тесты нужны для сборки и проверки, но не для runtime.

## Что делать после завершения работы

1. Прогнать `docker compose up --build` и убедиться, что все 6 контейнеров в `Up`.
2. Открыть `http://localhost:8080` и вручную проверить несколько CAPTCHA.
3. Зафиксировать для заказчика:
   - список `CHALLENGE_ID`;
   - пороги сложности `20 / 60 / 80 / 100`;
   - что задания server-driven и не решаются полностью на клиенте.
4. Передать заказчику либо весь исходник, либо release-артефакт с бинарем и встроенными HTML.
